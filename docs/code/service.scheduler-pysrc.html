  1  ''' 
  2  This is the application scheduler.  
  3  It defines scheduled tasks and runs them as per their defined schedule. 
  4   
  5  This scheduler is started and stopped when the app is started and stopped.  
  6  Unless RUN_SCHEDULE is set to False in the config. In which case it must be started manually / managed by supervisor. 
  7  It is presumed to run on one machine at present. 
  8   
  9  If scaling later requires having multiple machines, then this scheduler should only run on the machine that has access to  
 10  the relevant directories. There is a task that moves files from ftp user jail directories to tmp processing locations, and  
 11  this is the limitation - creating sftp accounts has to happen on one machine or across machines, but that would increase  
 12  attack surface for security vulnerability. So probably better to have only one machine open to sftp, and if necessary for  
 13  later scale the script that is called to move data from the sftp jails to processing locations could do so by round-robin  
 14  to multiple processing machines. The jper app config has settings for running this scheduler and what frequencies to run each  
 15  process, so it is just a case of installing jper on each machine but only setting the frequencies for the processes desired to  
 16  be scheduled on each given machine. 
 17   
 18  Or, if scheduled tasks themselves also need to be scaled up, the scheduler can continue to run on  
 19  all machines but some synchronisation would have to be added to that tasks were not run on every machine. Also, each machine  
 20  running the schedule would need access to any relevant directories.
