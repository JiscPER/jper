Module Hierarchy

  • config: Directory for all the service-specific configuration
    • config.service: Main configuration file for the application
  • service: The root module for the entire application.
    • service.api: This is the main Python API for interacting with the JPER system.
    • service.control: Module which controls the active role of accounts
    • service.dao: This module contains all the Data Access Objects for models which are persisted to Elasticsearch at some point in their lifecycle.
    • service.forms: This module contains all the model objects used by the system core.
      • service.forms.adduser: Created on 18 Nov 2015
      • service.forms.webservice: Created on 18 Nov 2015
    • service.initialise: JPER service initialise module, run at application startup.
    • service.models: This module contains all the model objects used by the system core.
      • service.models.account
      • service.models.api: Model objects which are used in providing the Python API to JPER
      • service.models.contentlog
      • service.models.ezb: Model objects used to represent interactions with ezb items
      • service.models.harvester
      • service.models.notifications: Model objects used to represent system core notification objects
      • service.models.repository: Model objects used to represent interactions with repositories
      • service.models.xslt: A quasi-model object used to represent xml stylesheets (xslt) for package convertion tasks
    • service.packages: Provides general packaging handling infrastructure and specific implementations of known packaging formats
    • service.reports: Functions which generate reports from the JPER system
    • service.routing: Module which handles all the routing mechanics to convert UnroutedNotifications into either RoutedNotifications or FailedNotifications
    • service.routing_deepgreen: Module which handles all the routing mechanics to convert UnroutedNotifications into either RoutedNotifications or FailedNotifications
    • service.routing_deepgreen_debug: Module which handles all the routing mechanics to convert UnroutedNotifications into either RoutedNotifications or FailedNotifications
    • service.scheduler: This is the application scheduler. It defines scheduled tasks and runs them as per their defined schedule.
    • service.tests: This module contains all of the system tests and associated fixtures
      • service.tests.fixtures: This module contains all of the fixtures that are used throughout the testing.
        • service.tests.fixtures.api: Fixtures for testing the API
        • service.tests.fixtures.notifications: Fixtures for testing notifications
        • service.tests.fixtures.packages: Fixtures for testing packages
        • service.tests.fixtures.repository: Fixtures for testing repository features
      • service.tests.functional: Module which contains tests and scripts for functional system testing
        • service.tests.functional.clear_test_data
        • service.tests.functional.harness: Script which can be used to drive all aspects of the JPER API in order to test its behaviour at scale
        • service.tests.functional.load_route_zip: Script which can be used to load a ZIP file into the system for the purposes of testing
        • service.tests.functional.load_test_data: Script which can load repository configs and user accounts into an empty test system, to prime it for the test harness
        • service.tests.functional.random_repos
        • service.tests.functional.test_api: Functional tests for the API.
      • service.tests.unit: System unit tests
        • service.tests.unit.test_api: Unit tests for the JPER API class
        • service.tests.unit.test_dao: Unit tests for the Data Access Objects
        • service.tests.unit.test_models: Unit tests for the core system objects
        • service.tests.unit.test_packager: Unit tests for the packaging system
        • service.tests.unit.test_report: Unit tests for the monthly report
        • service.tests.unit.test_report_debug: Unit tests for the monthly report
        • service.tests.unit.test_routing: Unit tests for the routing system
    • service.views: This module contains all the blueprints that are loaded by the application to provide the web interface
      • service.views.about: Created on 18 Nov 2015
      • service.views.account: Blueprint for providing account management
      • service.views.harvester: Created on 18 Nov 2015
      • service.views.more: Created on 18 Nov 2015
      • service.views.query: An elasticsearch query pass-through. Has auth control, so it is better than exposing your ES index directly.
      • service.views.reports: Blueprint for providing reports UI
      • service.views.test: Blueprint for providing test endpoints for when the system is run in functional test mode.
      • service.views.webapi: Blueprint which provides the RESTful web API for JPER
    • service.web: Main module from which the application is started and the web interface mounted
